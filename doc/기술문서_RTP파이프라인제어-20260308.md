# WebRTC A/V PTT 초저지연 및 배터리 최적화 아키텍처 명세서

_author: kodeholic (powered by Gemini)_

## 1. 개요 및 목적

본 문서는 `oxlens-sdk-core` 프로젝트의 모바일 환경에서 1) LTE/5G 모뎀 배터리 소모 방지, 2) 오디오 발언권 획득 시 0ms 지연 달성, 3) 비디오 카메라 콜드 스타트 극복을 동시에 만족하기 위한 클라이언트-서버 하이브리드 제어 아키텍처를 정의합니다.

## 2. 핵심 아키텍처 전략: Audio / Video 제어 분리 (Decoupling)

미디어 특성과 하드웨어 디코더의 민감도를 고려하여 오디오와 비디오의 대기(Idle) 상태 제어 정책을 완벽히 분리합니다.

- **Audio (0ms 응답성 최우선):** 클라이언트 내부 메모리 직접 주입(Dummy Injection) 및 패킷 변조 적용.
- **Video (안정성 최우선):** 클라이언트 패킷 조작 완전 배제(Over-engineering 방지). 서버 주도의 다중 Keyframe 요청으로 제어.

---

## 3. Audio 파이프라인: 로컬 더미 주입 및 NAT 패턴

오디오(NetEQ)는 패킷 유입이 중단되면 지터 버퍼가 비워지며 콜드 스타트 지연(첫 단어 잘림)이 발생합니다. 이를 막기 위해 네트워크 소켓을 우회하는 인터셉터(Interceptor)를 배치합니다.

### 3.1 동작 시나리오

1. **대기 상태 (Floor Idle):**
   - 서버와의 실제 RTP 통신 중단 (모뎀 수면 모드 진입).
   - Rust FSM 타이머가 200ms 주기로 C++ 래퍼를 호출하여 메모리 상에서 가짜 무음 패킷(PT=13) 조립.
   - 네트워크/복호화 레이어를 건너뛰고 엔진 수신부(`PacketReceiver`)에 직접 주입. NetEQ 핫(Hot) 상태 유지.
2. **발언권 획득 (Floor Taken) - 실제 패킷 유입:**
   - 실제 첫 패킷 도착 시, 로컬 더미 패킷과의 Sequence/Timestamp 차이(Offset) 계산.
   - 이후 유입되는 모든 실제 패킷의 헤더에 Offset을 더하여(Rewriting) NetEQ로 전달. Jitter Buffer Flush 없이 매끄러운 오디오 재생 보장.

### 3.2 아키텍처 다이어그램

```text
[Network Socket] --> [SRTP Decrypt] --> [ RtpPacketRewriter (Hook) ] --> [NetEQ (Audio Buffer)]
                                                  ^
[Rust Core FSM] --(FFI System Call)---------------+ (오직 Audio SSRC에만 개입)
```

## 4. Video 파이프라인: 서버 주도의 하드웨어 콜드 스타트 극복

비디오에 임의의 가짜 H264/VP8 패킷을 주입하면 모바일 OS의 하드웨어 디코더(MediaCodec) 파싱 에러 및 Fatal Crash를 유발합니다. 따라서 클라이언트 클라이언트 C++ 레이어에서는 비디오 패킷을 전혀 조작하지 않습니다.

### 4.1 카메라 하드웨어의 한계 (AE/AWB 딜레이)

발언권을 획득하여 송신자 카메라가 켜지더라도, 자동 노출(AE)과 화이트 밸런스(AWB)가 맞춰지는 데 물리적으로 300ms ~ 800ms가 소요됩니다. 즉시 전송된 첫 I-Frame은 흑백이거나 초점이 나간 쓰레기 데이터일 확률이 높습니다.

### 4.2 서버 주도 3중 Keyframe(PLI/FIR) 요청 시나리오

클라이언트의 뷰어 안정성을 위해, 발언권이 전환되는 시점에 **SFU 서버가 송신자에게 시차를 두고 총 3회의 Keyframe을 강제 요청**합니다.

- **1차 요청 (0ms):** 핫(Hot) 상태로 대기 중이던 카메라를 위한 즉각 반응용.
- **2차 요청 (500ms):** 하드웨어 노출/초점이 안정화된 직후의 '실질적인 첫 유효 프레임' 확보용.
- **3차 요청 (1200ms):** 저사양 기기 딜레이 및 네트워크 유실을 방어하는 최후의 보루(Safety Net).

**결론:** 수신 클라이언트는 멈춰 있던 화면에서, 서버가 조율해 준 깨끗한 Keyframe이 도착하는 순간 자연스럽게 화면을 갱신하게 됩니다.

## 5. 기술적 타당성 및 리스크 점검

1. **성능 영향 가능성 (Performance Impact):**
   - Audio 패킷 변조 시 16bit(Seq), 32bit(TS) 정수 덧셈 2회만 수행됩니다.
   - 상태 플래그 및 Offset 변수는 `std::atomic`을 활용한 Lock-Free 구조로 설계되어 미디어 스레드의 블로킹이 발생하지 않습니다.
2. **메모리 누수 가능성 (Memory Leak):**
   - Zero-Copy 원칙을 적용합니다. 엔진이 관리하는 `rtc::CopyOnWriteBuffer`의 원시 포인터를 참조해 헤더 바이트만 In-place로 덮어쓰므로 메모리 누수 위험을 원천 차단했습니다.
3. **확장 가능한 구조 (Scalability):**
   - `RtpPacketRewriter`는 오디오 SSRC 배열을 동적으로 주입받아 처리할 수 있어, 다중 채널(메인/서브 무전) 환경에서도 유연하게 대응 가능합니다.

## 6. 핵심 인터페이스 명세 (C++)

```cpp
// author: kodeholic (powered by Gemini)

#pragma once

#include "rtc_base/copy_on_write_buffer.h"
#include <atomic>
#include <cstdint>
#include <vector>

namespace oxlens {
namespace webrtc_ext {

constexpr uint8_t K_OPUS_CN_PAYLOAD_TYPE = 13;
constexpr uint32_t K_TIMESTAMP_STEP_20MS = 960;

class RtpPacketRewriter {
public:
    RtpPacketRewriter() :
        is_dummy_mode_(false), needs_offset_calc_(false),
        seq_offset_(0), ts_offset_(0), last_dummy_seq_(0), last_dummy_ts_(0) {}

    // [의도] 오디오 SSRC에 대해서만 더미 주입 및 변조 활성화 (비디오 배제)
    void StartAudioDummyInjection(uint32_t target_audio_ssrc);
    void PrepareForRealAudioStream();

    // 엔진 수신부 훅(Hook). 비디오 패킷은 조건문으로 Pass-through 처리
    rtc::CopyOnWriteBuffer RewritePacket(const rtc::CopyOnWriteBuffer& packet);

private:
    std::atomic<bool> is_dummy_mode_;
    std::atomic<bool> needs_offset_calc_;

    // Lock-Free 덧셈 연산을 위한 오프셋 변수 (Audio 전용)
    std::atomic<int32_t> seq_offset_;
    std::atomic<int64_t> ts_offset_;

    uint16_t last_dummy_seq_;
    uint32_t last_dummy_ts_;
    uint32_t active_audio_ssrc_;
};

} // namespace webrtc_ext
} // namespace oxlens
```
